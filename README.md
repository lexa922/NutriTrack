# NutriTrack — Додаток для контролю харчування
NutriTrack — це десктопний додаток на базі .NET WPF, створений для відстеження щоденного споживання калорій та балансу білків, жирів і вуглеводів (БЖВ). Проєкт реалізований з чітким дотриманням архітектурного патерну MVVM та принципів чистого коду.

## 🚀 Функціонал програми
- Авторизація та профілі: Розрахунок індивідуальної норми калорій залежно від параметрів користувача (вага, зріст, вік, активність).

- Журнал їжі: Додавання спожитих продуктів із зазначенням ваги.

- База продуктів: Пошук та фільтрація продуктів у реальному часі.

- Аналітика: Автоматичний розрахунок сумарних калорій та нутрієнтів за поточний день.

- Редагування: Можливість змінювати або видаляти записи в журналі.

## 🏛 Programming Principles  
У проєкті реалізовано наступні принципи програмування:

- Single Responsibility Principle (SRP): Кожен клас має одну зону відповідальності. Наприклад, SqliteNutriRepository відповідає тільки за доступ до даних, а CalorieCalculator — тільки за математичні розрахунки.

- Open/Closed Principle (OCP): Завдяки патерну "Стратегія", ми можемо додавати нові формули розрахунку калорій, просто створюючи нові класи без зміни логіки самого калькулятора.

- Dependency Inversion Principle (DIP): Високорівневі модулі (MainViewModel) не залежать від конкретних реалізацій (SQLite), а залежать від абстракцій (INutriRepository).

- DRY (Don't Repeat Yourself): Спільна логіка оновлення UI винесена в BaseViewModel, що позбавляє необхідності дублювати виклики PropertyChanged.

- Information Expert (GRASP): Обчислення калорій для конкретного запису їжі реалізовано всередині моделі FoodLog, оскільки саме вона володіє всіма необхідними для цього даними (вага та параметри продукту).

## 🔧 Refactoring Techniques
   Під час розробки та фіналізації коду були застосовані такі техніки рефакторингу:

- Extract Interface: Виділено інтерфейси для репозиторію та сервісів, щоб забезпечити слабку зв'язність модулів.

- Inject Dependency: Замість жорсткого створення об'єктів калькулятора всередині ViewModel, вони передаються через конструктор.

- Move Method: Логіка розрахунку калорій за порцію була перенесена з ViewModel безпосередньо в клас моделі FoodLog для кращої інкапсуляції.

- Extract Method: Метод InitializeAsync у MainViewModel був декомпонований на декілька приватних методів з чіткими назвами для покращення читабельності.

- Replace Magic Number with Symbolic Constant: Всі константи (наприклад, 100 грамів для розрахунків) були замінені на іменовані змінні.
